# 日志模块设计方案

> 历史记录（已完成，非现行事实源）：本计划记录独立日志模块的设计与落地；当前实现、测试清单和验证准入以 [测试规范](../design/testing/testing-specification.md) 和 [日志契约](../design/contracts/log-interface-protocol.md) 为准。

**目标：** 新增独立的结构化日志模块，承接 UI/业务/算法/HAL/Adapter 的 `logProduced` 事件，提供内存缓存、查询、持久化和 HAL 日志映射能力；不纳入参考目录中的 monitor UI 部分。

**架构定位：** 日志模块是五层架构旁路基础模块，不属于 HAL，也不反向依赖 UI/业务/算法。HAL 继续只生产 `HalLogEvent`，日志模块负责映射为项目主模型 `LogEvent`。

**参考取舍：**
- 参考 `Logger.h`：等级过滤、多输出目标、流式体验思想。
- 参考 `LoggerExtensions.h`：仅作为文本格式化灵感，不作为核心契约。
- 排除 `monitor.hpp`、`monitor_ui.*`、`monitor_ui_logger.*`：这些属于 monitor UI 桥接。
- 不直接移植：全局单例、`LOG_INFO` 全局宏、ANSI 颜色、同步 stdout/file 写入、`localtime_r`、回调持锁执行。

## 核心设计

1. 命名空间使用 `hwtest::logging`；HAL 仍使用 `hwtest::hal`。
2. `LogEvent` 保持现有契约字段：`timestampUs、level、source、category、message、requestId、durationMs、status、adapterCode、context`。
3. 增加 `LogLevel` 辅助枚举：`Trace、Debug、Info、Warn、Error、Fatal、Off`；`Off` 只用于过滤，不作为事件等级。
4. `ILogService` 提供：
   - `append(const LogEvent&)`
   - `recent(int maxCount) const`
   - signal `logAppended(const LogEvent&)`
5. `LogService` 负责：
   - 补齐缺省时间戳和等级。
   - 按最小等级过滤。
   - 维护有界 recent ring buffer。
   - 向 sink 分发日志，但不在内部状态锁内调用 sink 或 emit signal。
6. `JsonLineFileSink` 负责结构化落盘，每条日志一行 JSON，不写 ANSI 颜色。
7. `hal_log_bridge` 提供：
   - `LogEvent fromHalLogEvent(const hwtest::hal::HalLogEvent&)`
   - `QMetaObject::Connection connectHalLogs(hwtest::hal::IHalService*, ILogService*)`
8. HAL 映射规则：
   - `HalLogEvent.source` 为空时补 `hal`。
   - `deviceId/resourceId/operation` 写入 `LogEvent.context`。
   - `requestId/durationMs/status/adapterCode` 同时保留顶层字段和 context。
   - Adapter host log 仍先进入 HAL，再由 HAL 产生 `source="adapter"` 的 `HalLogEvent`。

## 实施步骤

1. 更新日志契约文档，明确 `LogEvent`、`ILogService`、等级、source、sink、HAL 映射和 monitor 排除边界。
2. 在根 `CMakeLists.txt` 增加 `add_subdirectory(src/logging)`，保持 Qt Core 查找逻辑不变。
3. 新建 `hwtest_log` 静态 Qt Core 库，公共 include 为 `src/logging/include`。
4. 实现 `log_types.*`：`LogEvent`、`LogLevel`、字符串转换、Qt metatype 注册。
5. 实现 `LogService`：线程安全 append/recent、等级过滤、有界缓存、`logAppended`。
6. 实现 `LogFormatter` 和 `JsonLineFileSink`：结构化 JSONL 输出，支持 flush，测试使用临时目录。
7. 实现 `hal_log_bridge.*`：链接 `hwtest_hal` 与 `hwtest_log`，只做映射和连接，不改 HAL ABI。
8. 新增 `tests/log` 目标 `hwtest_log_tests`，链接 `hwtest_log`、`hwtest_hal`、GoogleTest。
9. 保留并补强现有 HAL 日志测试；如需要，再新增 `tests/hal/hal_log_test.cpp` 验证 HAL 实际事件字段完整性。
10. 更新测试规范和实现设计报告，说明日志模块独立于 HAL、无 UI、无 monitor 依赖。

## 验证步骤

```powershell
cmake -S . -B build_vs -G "Visual Studio 17 2022" -A x64
cmake --build build_vs --config Debug --target hwtest_log_tests
cmake --build build_vs --config Debug --target hwtest_hal_tests
ctest --test-dir build_vs -C Debug --output-on-failure
cmake --build build_vs --config Release --target hwtest_log_tests
cmake --build build_vs --config Release --target hwtest_hal_tests
ctest --test-dir build_vs -C Release --output-on-failure
```

预期结果：

- `hwtest_log` 和 `hwtest_hal` 均可独立构建。
- `LogService` 过滤、recent 缓存、signal、JSONL 文件 sink 测试通过。
- `HalLogEvent -> LogEvent` 映射保留 `requestId、durationMs、status、adapterCode、deviceId、resourceId、operation`。
- 现有 HAL Mock 主流程不受影响。
- 仓库中不出现 `MB_DDF::Debug`、`LOG_INFO` 全局宏或 monitor UI 依赖作为主日志契约。
